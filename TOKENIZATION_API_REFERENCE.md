## Overview

### Purpose

The Inovio Tokenization Service allows merchants to:
- Send card PAN (Primary Account Number) to the tokenization service
- Receive a secure token in return
- Use the token in subsequent payment transactions
- **Tokens are single-use only** - a new token must be requested for each transaction

### Audience

This documentation is intended for:
- Technical staff at merchant organizations
- Developers integrating with the Inovio payment gateway
- Assumes working knowledge of:
  - HTTP POST/GET methods
  - XML/JSON data parsing
  - HMAC-SHA256 cryptographic signatures

### Security Requirements (Version 4.9+)

Clients using the enhanced tokenization service must:
- **Sign request packets** in headers for verification by Inovio
- **Verify response signatures** generated by Inovio
- Use **HMAC-SHA256 Base16 (hex)** encryption for all signatures
- Establish a **secret key** with Inovio gateway support

## Token Request

### Request URL

The tokenization service uses a **different endpoint** from the payment gateway:

```
https://api.inoviopay.com/payment/token_service.cfm
```

### Request Method

**HTTP GET** with query parameters and custom headers

### Header Request Parameters

| Field Name   | Description                                                                 | Format/Notes                                    | Version |
|--------------|-----------------------------------------------------------------------------|-------------------------------------------------|---------|
| X-TIMESTAMP  | Request timestamp (UTC, 5-minute tolerance window)                          | `YYYYMMDDHHmmss` (24-hour clock, UTC timezone) | 4.9+    |
| X-SIGNATURE  | Client's HMAC-SHA256 signature                                              | Hex string (Base16 encoding)                   | 4.9+    |

**Signature Generation:**
```
X-SIGNATURE = HMAC_SHA256_HEX(secret_key, X-TIMESTAMP + UNIQUE_ID + SITE_ID)
```

### Body (Query) Request Parameters

| Field Name              | Description                           | Required | Notes                    | Version |
|-------------------------|---------------------------------------|----------|--------------------------|---------|
| CARD_PAN                | Customer card number to tokenize      | Yes      | 13-19 digits             | All     |
| SITE_ID                 | Merchant's website ID                 | Yes      | Provided by Inovio       | 4.9+    |
| UNIQUE_ID               | Alphanumeric transaction identifier   | Yes      | Max 32 characters        | 4.9+    |
| REQUEST_API_VERSION     | API version from this document        | Yes      | e.g., "4.9" or "2.22"    | 4.9+    |
| REQUEST_RESPONSE_FORMAT | Response format (XML or JSON)         | No       | Default: JSON            | All     |

### Token Request Example

**Configuration:**
- Secret Key: `Password123`
- Site ID: `9201`
- Unique ID: `32123456789012345678901234567890`
- Timestamp: `20250129225200`

**Signature Calculation:**
```
message = "20250129225200" + "32123456789012345678901234567890" + "9201"
X-SIGNATURE = HMAC_SHA256_HEX("Password123", message)
// Result: 9cdc12c09f2fdb114167cad6fa451aeaa37c4df446720cae9805bf1e60e0603f
```

**Full Request:**
```http
GET https://api.inoviopay.com/payment/token_service.cfm?CARD_PAN=409159111111111&REQUEST_API_VERSION=2.22&UNIQUE_ID=32123456789012345678901234567890&SITE_ID=9201&REQUEST_RESPONSE_FORMAT=json
X-SIGNATURE: 9cdc12c09f2fdb114167cad6fa451aeaa37c4df446720cae9805bf1e60e0603f
X-TIMESTAMP: 20250129225200
```

| Field Name   | Description                                  | Format/Notes                                    | Version |

**Response Signature Verification:**
```
X-SIGNATURE = HMAC_SHA256_HEX(secret_key, X-TIMESTAMP + TOKEN_REQID + JSON_BODY)
```

| CARD_BANK                 | Issuer bank name for the BIN                     |                                    | 4.9+    |
| CARD_COUNTRY              | Issuer bank country                              | Three-character alpha code         | 4.9+    |

## API Error Codes

| Error Code | Description                                                  |
|------------|--------------------------------------------------------------|
| 121        | Request field X-Signature failed to be matched by Inovio     |
| 406        | Request field X-Timestamp is outside of 300-second range     |
| 502        | Request field SITE_ID secret key failed to be matched        |

### Error Handling Best Practices

1. **Error 121 (Signature Mismatch)**:
   - Verify signature generation algorithm
   - Check message concatenation order
   - Ensure secret key is correct
   - Confirm hex encoding (lowercase)

2. **Error 406 (Timestamp Out of Range)**:
   - Synchronize server clock with NTP
   - Use UTC timezone
   - Generate timestamp immediately before request

3. **Error 502 (Invalid Site ID or Secret)**:
   - Verify SITE_ID with Inovio support
   - Confirm secret key matches provisioned value
   - Check for extra whitespace or encoding issues

## Applied Use of Tokenization

### Common Integration Pattern

The most common application uses a web form with AJAX:

1. **User enters card number** in form field
2. **On blur event** (user tabs away), trigger token request
3. **JavaScript calls server-side endpoint** (never client-side!)
4. **Server generates signature** and requests token
5. **Server validates response signature**
6. **Token stored in hidden form field**
7. **Form submits** to payment gateway with token

## Sample Code

### Security Warning

⚠️ **CRITICAL:** The sample code in the original documentation shows client-side JavaScript signature generation. This is **NOT SECURE** and should **NEVER** be used in production.

**Correct Approach:**
- Signature generation must happen **server-side**
- Secret key must **never** be exposed to the browser
- Client-side code should only call your server endpoint
- Your server handles all cryptographic operations

### Server-Side Implementation (Node.js Example)

See the complete implementation in this repository:
- `server.js` - Full Express server with signature generation
- `public/index.html` - Secure client-side form
- `.env` - Configuration (never commit to git!)

**Key Functions:**

```javascript
// Generate UTC timestamp
function generateTimestamp() {
  const now = new Date();
  const year = now.getUTCFullYear();
  const month = String(now.getUTCMonth() + 1).padStart(2, '0');
  const day = String(now.getUTCDate()).padStart(2, '0');
  const hours = String(now.getUTCHours()).padStart(2, '0');
  const minutes = String(now.getUTCMinutes()).padStart(2, '0');
  const seconds = String(now.getUTCSeconds()).padStart(2, '0');
  return `${year}${month}${day}${hours}${minutes}${seconds}`;
}

// Generate HMAC-SHA256 signature
function generateSignature(secretKey, message) {
  return crypto
    .createHmac('sha256', secretKey)
    .update(message)
    .digest('hex');
}

// Verify response signature
function verifyResponseSignature(secretKey, receivedSignature,
                                  serverTimestamp, tokenRequestId,
                                  responseData) {
  const dataString = JSON.stringify(responseData);
  const messageToVerify = `${serverTimestamp}${tokenRequestId}${dataString}`;
  const expectedSignature = generateSignature(secretKey, messageToVerify);
  return expectedSignature.toLowerCase() === receivedSignature.toLowerCase();
}
```

## Security Implementation

### Request Signature Generation

**Algorithm:** HMAC-SHA256

**Message Format:**
```
timestamp + uniqueId + siteId
```

**Example:**
```javascript
const secretKey = "Password123";
const timestamp = "20250129225200";
const uniqueId = "32123456789012345678901234567890";
const siteId = "9201";

const message = timestamp + uniqueId + siteId;
// "2025012922520032123456789012345678901234567890" + "9201"

const signature = HMAC_SHA256_HEX(secretKey, message);
// Result: hex string (64 characters)
```

**Implementation (Node.js):**
```javascript
const crypto = require('crypto');

function generateRequestSignature(secretKey, timestamp, uniqueId, siteId) {
  const message = `${timestamp}${uniqueId}${siteId}`;
  return crypto
    .createHmac('sha256', secretKey)
    .update(message)
    .digest('hex');
}
```

### Response Signature Verification

**Algorithm:** HMAC-SHA256

**Message Format:**
```
serverTimestamp + tokenRequestId + rawJSONBody
```

**Example:**
```javascript
const secretKey = "Password123";
const serverTimestamp = "20241209230307";
const tokenRequestId = "4368675";
const rawJSONBody = '{"TOKEN_GUID":"C5178FCA...","CARD_BRAND_NAME":"Visa",...}';

const message = serverTimestamp + tokenRequestId + rawJSONBody;

const expectedSignature = HMAC_SHA256_HEX(secretKey, message);

if (expectedSignature.toLowerCase() === receivedSignature.toLowerCase()) {
  console.log("✅ Signature verified");
} else {
  console.error("❌ Signature mismatch!");
}
```

**Implementation (Node.js):**
```javascript
function verifyResponseSignature(secretKey, receivedSignature,
                                  serverTimestamp, tokenRequestId,
                                  responseData) {
  // Reconstruct exact JSON string from response
  const dataString = JSON.stringify(responseData);

  // Build message to verify
  const messageToVerify = `${serverTimestamp}${tokenRequestId}${dataString}`;

  // Generate expected signature
  const expectedSignature = crypto
    .createHmac('sha256', secretKey)
    .update(messageToVerify)
    .digest('hex');

  // Compare (case-insensitive)
  return expectedSignature.toLowerCase() === receivedSignature.toLowerCase();
}
```

### Security Best Practices

✅ **DO:**
- Keep secret key secure on server-side only
- Use environment variables for configuration
- Validate response signatures before trusting data
- Use HTTPS for all API communication
- Generate unique IDs for each request
- Synchronize server time with NTP
- Log signature verification failures for security monitoring

❌ **DON'T:**
- Never expose secret key in client-side code
- Never commit secret keys to version control
- Never skip signature verification
- Never reuse tokens (single-use only)
- Never trust timestamps older than 5 minutes
- Never send card PAN to payment gateway (use token instead)

### Debugging Signature Issues

**Common Problems:**

1. **Timestamp Format**
   - Must be UTC, not local time
   - Format: `YYYYMMDDHHmmss` (no separators)
   - Example: `20250129225200` not `2025-01-29 22:52:00`

2. **Message Concatenation**
   - No separators between values
   - Exact order: `timestamp + uniqueId + siteId`
   - Case-sensitive for all values

3. **Hex Encoding**
   - Must be lowercase hex characters
   - 64 characters long (256 bits / 4 bits per char)
   - No `0x` prefix

4. **Response Verification**
   - Use exact JSON string from response body
   - Don't re-serialize or pretty-print
   - Include all whitespace exactly as received

**Debug Script:**
```javascript
// Log all signature components
console.log('Timestamp:', timestamp);
console.log('Unique ID:', uniqueId);
console.log('Site ID:', siteId);
console.log('Message:', timestamp + uniqueId + siteId);
console.log('Secret Key:', secretKey);
console.log('Generated Signature:', signature);
```
